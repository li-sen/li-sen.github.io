什么是docker

用大白话说，容器技术就是把应用程序和操作系统打成一个包，然后在Docker引擎中可以运行这个容器。比如把Mysql和CentOS打成一个包，然后在Docker引擎中去运行。容器提供了运行应用所必须的所有资源，有应用本身和操作系统，这是和传统应用程序最根本的区别。这样的好处是，应用容器化之后，可独立运行，可快速移植，可快速启动，可快速扩容，标准化，可复制。特别是和微服务结合时，有更好的效果。

一个使用容器的场景 看了上面的描述，大家估计还是一脸蒙圈，不知道容器是什么东西。我会举个栗子，让大家对容器有个直观的了解。

要达成的目的：我们要运行一个Mysql，然后可以在命令行和应用中连接mysql

传统方式(非容器方式)：

去Mysql官网下载mysql的二进制文件，保存到本地机器某个路径，找到mysql的配置文件，进行修改，然后启动mysql 使用包管理器来安装mysql。Mac上的Homebrew，SDKMan，Redhat/CentOS上的yum，Ubuntu/Debian上的apt-get。Win上请自己开发。。。。。 传统方式，基本就是上面的两种方式来安装和使用Mysql了。我在Mac，CentOS上，这两种方式都试过，说下这种方式的弊端吧：

Mac上安装Mysql是使用Mac专有的pkg文件来安装的，安装好之后，会在Mac的系统面板中多一个图标。启动和停止Mysql都只能进去系统面板，点击启动停止按钮来停止Mysql。而且使用pkg安装的Mysql很难卸载，如果想安装新版本的Mysql，要卸载现有的Mysql，很麻烦。 Mac上使用Homebrew安装Mysql，如果自己玩玩，用Homebrew问题也不大，但homebrew安装后的文件目录和官方的Mysql是不一样的，比如mysql的配置文件路径都不一样，还必须要通过homebrew的info命令来查看安装信息，看配置文件在哪个路径。而且homebrew默认安装homebrew仓库的最新版本，如果要安装一个特定版本，比如Mysql5.5.38，请百度，比较的麻烦。 CentOS上安装Mysql，如果是下载二进制文件来安装，不同的人有不同的习惯，比如有人去官网下载Mysql，有人会去什么论坛下载一个不知道改了什么东西的Mysql,之前就有人这么干过。然后保存到本机的目录也是每个人都不一样，导致，A安装的Mysql，让B来管理，完全不知道怎么弄。 CentOS上使用yum安装Mysql，一条命令可以安装，但是如果安装特定版本的mysql，是比较麻烦的。而且，使用yum安装时很可能会失败，特别是安装的组件依赖了一些系统级别的gcc库，需要本地编译。或者在安装别的软件时，和mysql的某些文件不兼容。 通过上面的描述，这些问题是很常见的，就是因为我们的应用严重依赖它所处的环境，也就是操作系统。操作系统上安装有一些系统级别的库文件，多个应用程序之间会互相干扰。如果应用可以在一个隔离的沙盒中去运行，不受操作系统和其他应用的影响，会简单，可靠很多。

下面来看看怎么使用容器来运行一个Mysql

准备一个电脑，能开机就可以了，docker支持Windows，Unix，Linux等操作系统，几乎支持所有的操作系统。 安装Docker引擎，不同操作系统安装Docker引擎的方式不同，但都比较简单，和安装JVM虚拟机类似。 命令行运行 docker run -p 3306:3306 mysql 这样，一个Mysql就可以运行了。可以向传统方式一样来用命令行和jdbc连接字符串来访问mysql了。

如果不想使用mysql了，docker stop mysql即可。

如果想使用特定版本的mysql,docker run mysql:5.5.38即可。

如果想启动多个mysql，启动多个容器就可以了，多个mysql是完全隔离的。比如，运行3个相同版本5.5.38的mysql都是可以的。

如果想删除mysql，docker rm mysql即可。

镜像 上面的命令docker run mysql:5.5.38中，使用docker引擎运行了一个mysql的容器，容器使用的镜像是mysql，存储在官方的docker hub仓库。版本是5.5.38。镜像就是把应用，文件和操作系统打成的包。比如把Mysql和CentOS打成一个包。镜像在逻辑是一个文件，但物理上是分成多个文件层的。Docker的文件储存是分层的，每个层类似于git的一次commit。镜像有官方的，也可以自己制作镜像。镜像是不可变的。

容器 使用一个镜像可以运行一个或多个容器。一个容器就是一个进程，是隔离的。基于linux的cgroup做的隔离。镜像类似于Java里面的类，容器类似于类对应的实例。

容器数据存储 容器中运行的mysql，会产生数据，这些数据保存在哪里？容器停止或被删除后，这些数据还在吗？

容器中的应用产生的数据是保存在容器里面的，在本地机器的某个临时户目录，以docker的格式保存的，一旦容器被删除了，这些文件就被删除了。如果想对容器中的数据进行持久化，比如讲mysql的data文件写入本机的某个目录，就要用到docker中的数据卷，即volume。

docker run -v /some/path:/var/lib/mysql mysql。这条命令是将本机的/some/path目录挂载到容器里面的/var/lib/mysql目录，也就是myslq保存数据的目录。这样当容器里面的mysql写数据时，就向本地的目录写了。

容器中的网络 docker容器是一个进程，是互相隔离的，如何在容器外部访问容器里面的应用呢？比如访问容器里面的mysql服务。

docker run -p 3306:3306 mysql。大家注意到命令里有一段参数 -p 3306:3306。这个参数的意思是将容器里面的3306端口映射到本地机器的3306端口，这样，通过访问本机机器的3306端口，会转发到容器的3306端口。

那容器里机器的ip是多少呢？这就涉及到docker的网络模型。

bridge：默认模式，每个容纳有自己的网络，ip是内部ip172.17.0.X。 Host模式：容器和host机器的网络配置是一样的。ip也是一样的，但这样违背了容器的隔离性。多个容器的端口可能会冲突，而且，当host机器的网络比较复杂时，容器里面的网络也会发生一些不可预知的问题。 自定义模式，docker使用的是eth0网络设备，可以自定义docker的网络，k8s就是使用自定义的网络。 使用哪种网络视具体情况而定，比较简单的应用可以使用breidge模式，一些情况下可能要用到Host模式，比如在容器中运行dubbo，如果使用的是bridge模式，dubbo向zookeeper注册的ip地址是docker的内外地址，也就是172.17.0.X，导致容器外部的dubbo应用无法调用容器里面的服务。也就是说所有的dubbo容器必须在同一台机器上。如果垮多台机器，无法互相调用。这时候就必须使用Host模式了。

可见，docker自带的网络模型是很弱的，跨Host的容器之间是不能通信的，无法用于实际的生产环境中。必须要依靠一些专门的容器编排工具，比如下面介绍的Kubernetes。

docker 中文意思为 码头工人，有点类似集装箱的概念，简单来讲就是 容器虚拟化的一种实现，主要利用linux内核中的Namespaces 以及Cgroup，进行内核级别的资源隔离以及限制，用户操作 Docker 的容器就像操作一个快速轻量级的虚拟机一样简单。 > 关于namespace以及cgroup 做下简单说明，如果需要深入 请自行谷歌百度 - Namespace ：简单来讲就是可以实现每个容器只能看到和使用它自己的资源(资源隔离)。

docker vs vm

docker 也是虚拟化技术的一种，不过它比传统的vm更加轻量、快速， 更关键的是他能提供一个统一的、可控的软件环境，有点类似jvm 跨平台一样，一次编译到处运行；这使得软件的开发以及交付 对于vm是一种质飞跃。

一个容器一个程序（进程）

容器有个默认规范或约定，就是一个容器 只运行一个程序或进程，这是一个很nice的设计哲学，这体现在：

  1. 安全隔离：一个程序用一个容器，挂了、被入侵了可以很好的隔离，对其他程序无影响
  2. 最小粒度控制资源:
    1. 最少化使用资源：哪些程序需要资源多，提高对应容器的资源即可，不用一锅粥的提升
    2. 更好的管理： 比如扩缩容nginx 副本数，只要对nginx容器进行副本数增加即可

这些都是一个容器跑多个程序做不到，不过这仅仅是我个人的一些见解。

docker 运行原理

其实容器虚拟化技术很早就有了，那为什么现在才突出重围，docker做了哪些改变才是容器爆发出来？

docker 最大的突破是引入镜像技术，可以让开发者轻松打包他们的应用以及相关的环境依赖生成docker镜像，然后利用镜像仓库可以快速发布到任何支持docker的 主机上，开发者不用再关注部署环境主机硬件、系统版本、运行环境等等环境配置问题，只要有docker服务，docker run 就解决了，很好解决了环境一致性问题，从而极大的提高了软件开发交付效率！

docker运行架构

docker 也是c/s架构 ，用户通过docker client 与docker daemon 通信，docker daemon 接收请求后，查询本地是否有对应镜像，如果有就使用本地镜像，启动 对应容器，如果没有就去指定 Registry 仓库拉取，然后存在本地，再执行启动步骤。

镜像技术 ：
